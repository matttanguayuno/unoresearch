AI Skills

Modular, reusable AI capability bundles that agents invoke on demand - the first “productizable” layer of AI customization.

1. What AI Skills Are

Skills are modular, reusable AI capability bundles invoked on demand by an agent. Unlike prompts or instructions (always loaded), skills are selectively loaded - pulled in only when the agent encounters a matching task. This makes them the most structured layer of AI customization and the first that can be packaged, shared, and sold.

2. Skill File Structure

Every skill follows the Agent Skills specification - a folder containing a SKILL.md file plus optional supporting directories.

Directory Layout

your-skill-name/
├── SKILL.md          # Required - main skill file
├── scripts/          # Optional - executable code
│   ├── process_data.py
│   └── validate.sh
├── references/       # Optional - documentation
│   ├── api-guide.md
│   └── examples/
└── assets/           # Optional - templates, etc.
    └── report-template.md

Naming rules

  • File must be exactly SKILL.md (case-sensitive)
  • Folder name: kebab-case only (e.g. notion-project-setup)
  • No spaces, underscores, or capitals in folder name
  • Folder name must match the name field in frontmatter

YAML Frontmatter

The frontmatter is how the agent decides whether to load your skill. Only name and description are required.

---
name: pdf-processing
description: Extract PDF text, fill forms,
  merge files. Use when handling PDFs.
license: Apache-2.0
compatibility: Requires Python 3.14+ and uv
metadata:
  author: example-org
  version: "1.0"
allowed-tools: Bash(git:*) Read
---
FieldRequiredNotes
nameYesKebab-case, max 64 chars, must match folder name
descriptionYesMax 1024 chars - what it does and when to use it
licenseNoLicense name or reference to bundled file
compatibilityNoEnvironment requirements (product, packages, network)
metadataNoArbitrary key-value pairs (author, version, etc.)
allowed-toolsNoPre-approved tools list (experimental)

Progressive Disclosure

Skills are loaded in three tiers to minimize context usage:

1
Metadata ~100 tokens

name + description loaded at startup for all skills

2
Instructions < 5 000 tokens

Full SKILL.md body loaded when the skill is activated

3
Resources as needed

Files in scripts/, references/, assets/ loaded only when required

4. Packaging, Marketing & Repositories

Skills are being positioned as the first productizable layer of AI - portable expertise modules that give general-purpose agents specialized, professional-grade workflows. The shift is from disposable prompts to reusable, versioned assets.

📦

Packaging: “Everything is a Folder”

  • Core file - A SKILL.md with structured instructions, constraints, and examples. Metadata (name + description) lets agents decide relevance before loading the full body.
  • Bundled assets - Optional subdirectories for helper scripts (Python, Bash), reference docs, templates, and data files needed to complete tasks.
  • Distribution formats - Plain directories on GitHub, zipped packages, CLI plugin installs (copilot plugin install, /plugin install), or dedicated registries.
📩

Marketing as “Packaged Expertise”

  • Methodology over instruction - Creators market skills as codified personal or institutional methodology, not just prompts. E.g. “50+ skills that chain a full marketing campaign.”
  • Specialized packs - Bundled into functional roles: Product Marketing Manager Skills Pack (battlecard generators, launch playbooks), Security Audit Pack, etc.
  • B2B / enterprise integration - Partners like Atlassian, Stripe, and Notion ship official “agent-first” documentation as skills, grounding agent behavior in authoritative product data and reducing hallucinations.
🚀

7. Risks & Limitations

Skills are still early. Key challenges to watch:

!

Security Surface

Skills instruct agents to create files, run commands, and call APIs. A poorly-written or malicious skill is a real attack vector. No vendor has a review/signing model yet.

!

Skill Conflicts

When multiple skills match the same context, behavior is unpredictable. Priority, ordering, and override rules are not standardized across any tool.

!

Versioning & Maintenance

Skills drift as codebases evolve. No tool tracks whether a skill is stale or contradicts current project conventions. Maintenance burden is invisible.

!

Debugging & Observability

When a skill misfires, it is hard to tell which skill was loaded, why it triggered, or what it injected. Tracing and logging are minimal across all vendors.

!

Governance

Enterprise teams need to control which skills agents can use, who can publish them, and audit what they do. Only GitHub Copilot has early org-level controls.

Documentation is changing

Docs are not just for humans anymore. They now need to work for people, LLMs, and agents/tools too.

The short version

The biggest shift is that we should stop thinking about everything as just "docs". There are really a few different layers now:

📖

Traditional Docs

Help humans understand things

▾

Still the foundation. People need tutorials, how-to guides, reference pages, and explanation. This part has not gone away.

🤖

LLM-Ingestible Docs

Help models retrieve the right context

▾

Documentation structured so models can use it well. The key idea is not "AI magic" - it is mostly clean structure, good headings, self-contained sections, and markdown or plain-text outputs.

🔧

Tools

Let models take actions

▾

A tool is not just a page explaining a feature - it is something callable, with clear inputs, outputs, and expected actions. Tool documentation needs to be much more explicit than normal prose.

💬

Prompts

Guide how a model approaches a task

▾

Reusable guidance - useful for framing, templates, and repeatable instructions. But usually not the best long-term place to store important operational knowledge.

⚡

Skills

Package repeatable workflows or playbooks

▾

Probably the most interesting layer. A skill is a reusable playbook the model can load when needed - a great fit for repeated internal workflows, conventions, and step-by-step processes.

Why this matters

Old model

Write docs pages for humans

→
New model

Write once, then publish in a few ways depending on the job

So the same source material might become:

  • A normal docs page for people
  • A markdown or LLM-friendly version for retrieval
  • A tool schema for action
  • A skill for repeatable workflows

The real shift is not just "make docs AI-friendly".

It is: separate context, actions, and workflows clearly.

When those are separated:

📄 Docs are easier to read
🔍 Retrieval gets cleaner
🛡️ Tools become safer and more reliable
🔁 Repeated processes are easier to automate
